# WebP Converter

A small desktop app that wraps Google's official `cwebp` encoder so anyone on the team can bulk-convert images to WebP without touching the terminal. PNGs go lossless (transparency preserved); JPG/TIFF/BMP use a tunable quality slider.

This is the GUI version of [../convert-to-webp.sh](../convert-to-webp.sh). The script and the app produce the same output — the app just packages it for non-engineers.

## What teammates need to do to use it

**Nothing — `cwebp` is bundled inside the app.** They install the `.dmg` (Mac) or `.msi` (Windows), open the app, pick a folder, click Convert.

## Features

- Pick input + output folders with native dialogs
- **Auto-detect per file:** PNG → lossless, everything else → quality slider (default 80)
- Quality slider (0–100), parallel jobs (1–32), include subfolders, skip already-converted
- Live progress bar and per-file ticker
- Final summary: total saved, % saved, list of failures

## Building from source

### Prerequisites

| Tool | Why | Install |
|---|---|---|
| Rust (stable) | Tauri backend | `curl --proto '=https' --tlsv1.2 -sSf https://sh.rustup.rs \| sh` |
| Node 18+ | Tauri CLI | https://nodejs.org or `brew install node` |
| Xcode CLT (macOS) | Linker / SDK | `xcode-select --install` |
| Visual Studio Build Tools (Windows) | Linker / SDK | https://visualstudio.microsoft.com/downloads/ |

### One-time setup

```bash
cd webp-converter-app
npm install
```

### Run in dev mode (hot reload)

```bash
npm run dev
```

### Build a distributable

```bash
npm run build
```

Output lands in `src-tauri/target/release/bundle/`:

- **macOS:** `dmg/WebP Converter_0.1.0_<arch>.dmg`
- **Windows:** `msi/WebP Converter_0.1.0_x64_en-US.msi`

Note: each platform's installer must be built **on that platform** (Tauri does not cross-compile by default). The Windows `.msi` has to come from a Windows machine or CI.

### Building for both Mac architectures

By default `npm run build` produces an installer for whatever Mac you're on (arm64 on Apple Silicon, x86_64 on Intel). To build for the *other* architecture from an Apple Silicon Mac:

```bash
rustup target add x86_64-apple-darwin
npm run build -- --target x86_64-apple-darwin
```

For a universal binary that works on both:

```bash
rustup target add x86_64-apple-darwin aarch64-apple-darwin
npm run build -- --target universal-apple-darwin
```

## Code signing & notarization (macOS)

Without an Apple Developer cert, teammates will see a Gatekeeper warning the first time they open the app. They can bypass it once by **right-clicking the app → Open → Open**, after which macOS remembers.

If you have an Apple Developer ID, signing is handled by setting these env vars before `npm run build` — see https://tauri.app/distribute/sign/macos/ for the full flow.

## How the bundled cwebp works

The three `cwebp` binaries in [src-tauri/binaries/](src-tauri/binaries) are Tauri *sidecars* — they get copied into the app bundle at build time and resolved at runtime via the target triple suffix in their filename:

- `cwebp-aarch64-apple-darwin` (Apple Silicon Macs)
- `cwebp-x86_64-apple-darwin` (Intel Macs)
- `cwebp-x86_64-pc-windows-msvc.exe` (64-bit Windows)

All three are from [Google's libwebp 1.5.0 release](https://developers.google.com/speed/webp/download). To upgrade, replace the files with new ones (preserve the filename suffixes).

## Architecture

The whole app is two Tauri commands and one event:

- **`scan(input_dir, recursive) → count`** — walks the folder, counts supported images.
- **`convert(options) → ConvertSummary`** — walks the folder, picks lossless-vs-quality per file based on extension, spawns up to N parallel `cwebp` sidecar processes (gated by a `tokio::Semaphore`), and emits a `convert:progress` event after every file completes.

The frontend listens to `convert:progress`, updates the bar and the per-file ticker, then renders the final summary card when the `convert` promise resolves.

## Troubleshooting

**"This app cannot be opened" on macOS** — right-click → Open → Open. First-launch Gatekeeper prompt, not a real error.

**`cwebp not found` at runtime** — only happens if the sidecar wasn't bundled correctly. Confirm `src-tauri/binaries/cwebp-<triple>` exists for your platform before building. Tauri logs the resolved path during a dev run.

**Builds for the wrong CPU** — pass `--target <triple>` to `npm run build` (see above).

**Some files fail with "decoding error"** — usually a corrupt source or CMYK JPEG with an exotic profile. The summary card lists every failure.
